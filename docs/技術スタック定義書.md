## 2. 永続層：Convex（セルフホストOSS）

- **セルフホスト**：クラウド版と同一のOSSバックエンドコードを自前サーバーで稼働可能。ADR-5（セルフホスト優先・単一テナント）を満たす。
- **関数モデル**：
  - `query` … 読み取り専用・**リアクティブ**（購読クライアントへ自動push）
  - `mutation` … 書き込み・**トランザクション**（楽観的並行制御＝OCC、競合時は自動リトライ）
  - `action` … 外部I/O可（非トランザクション）。外部API呼び出しやWebhookの副作用処理に使う
- **スキーマ**：`convex/schema.ts` の `defineTable` ＋ バリデータ（`v.string()`, `v.id()`, `v.union(v.literal(...))`）で §3 の論理モデルを表現する。

## 3. 基本設計との対応（INVARIANT / ADR → Convex機能マッピング）

本書の核心。基本設計で「機構を問わない」とした不変条件を、Convex のどの機能で満たすかを確定する。

| 基本設計の項目 | Convex での実現 |
|---|---|
| **INVARIANT-1 採番の一意性** | カウンタ用ドキュメントを `mutation` 内で読み取り→インクリメント→書き込み。mutation はトランザクション＋OCC自動リトライのため、同時実行でも `number` の重複は発生しない |
| **INVARIANT-2 並行更新の検出** | Convex の OCC が競合 mutation を自動リトライ。さらに reactive query で UI が常に最新化されるため stale write 自体が減る。明示的な「他の人が編集中」UXが要る場合のみ `revision` フィールドを併用 |
| **INVARIANT-3 参照整合性** | スキーマで `v.id("tasks")` 等の型付き参照を使い、`mutation` ロジック内で実在検証。DBレベルのFK制約に依存しない（基本設計の論理モデルどおり） |
| **INVARIANT-4 状態の妥当性** | §5 の状態機械を `mutation`（`transitionStatus`）内に単一実装。UI・MCP・Webhook の全経路がこの mutation を通る |
| **INVARIANT-5 Issue–Task の最低基数と整合** | `createIssue` mutation は最初の Task を同一トランザクションで作成（≥1 Task を保証）。`Task.project` は `issue.project` から導出して保存。Issue の派生ステータスは `lib/issueStatus.ts`（§5.1）で Task 群から算出し、`transitionStatus` 等の Task 変更時に再計算する |
| **ADR-7 同期方式** | `query` のリアクティブ購読が WebSocket層を代替。Convex React client が楽観的更新も担う。自前の同期層実装が不要になる |
| **ADR-1 MCPは永続層に直接触れない** | MCPサーバーは UI と同じ Convex 公開関数を呼ぶ。不変条件は関数側で強制され、3経路で一貫する |
| **CRDTの隔離（ADR-7）** | UIは reactive query 抽象に依存。将来CRDTが必要になっても、データモデル・関数インターフェースを保ったまま同期層のみ差し替え可能 |

> **効果**：ADR-7 で「自前実装すると重い」と評価した購読・トランザクション・並行制御の3点が、すべてプラットフォーム標準で満たされる。MVPの実装量が大幅に減る。

## 4. Coreロジック構成（Convex functions）

不変条件を集約する関数群。UI・MCP・Webhook はすべてここを経由する。

```
convex/
  schema.ts            … 論理モデルの物理定義（defineTable）
  issues.ts            … createIssue / listIssues / getIssue（mutation, query）。派生ステータスを付与
  tasks.ts             … createTask（issue 必須）/ updateTask / transitionStatus / assignTask / move（mutation, query）
  numbering.ts         … 採番（INVARIANT-1。Task.number / Issue.number）
  gitlinks.ts          … linkGit（mutation）
  projects.ts / members.ts
  http.ts              … Webルータ（httpAction：GitHub Webhook受信）
  crons.ts             … reconcile スケジュール
  lib/stateMachine.ts  … §5 状態遷移規則（transitionStatus が利用）
  lib/issueStatus.ts   … §5.1 Issue 派生ステータス算出（純粋関数。Task 群→status）
```

---

## 5. MCPサーバー（MVP中核）

- **形態**：独立した TypeScript プロセス。MCP公式SDK で Resources / Tools を公開。
- **接続**：Convex の HTTP API（`ConvexHttpClient`）経由で公開関数を呼ぶブリッジに徹する。**永続層に直接アクセスしない**（ADR-1）。
- **不変条件の強制**：採番・状態遷移・権限・監査は Convex mutation 側で担保されるため、MCPサーバーは薄く保てる。
- **Human-in-the-Loop**：`delete_task`・`canceled`遷移・一括操作は MCPホスト側で承認プロンプトを要求（基本設計 §6 準拠）。
- **アイデンティティ**：MCPアクセストークン → エージェント用 `Member` にマッピング（MVPはエージェント専用Member 1つ。on-behalf-of は Phase2）。

---

## 6. Git連携

- **Webhook受信**：`convex/http.ts` の `httpAction` でGitHubイベントを受ける。HMAC署名（`X-Hub-Signature-256`）を検証 → 内部 `mutation` を呼びステータス自動遷移（基本設計 §7）。
- **冪等化**：`X-GitHub-Delivery` を保存し重複処理を防止。
- **reconcile**：`crons` で定期的にGitHub APIをポーリングし、Webhook取りこぼしによるズレを補正（基本設計リスク#5）。`action` でGitHub API呼び出し、結果を `mutation` で反映。

---

## 7. フロントエンド

- **React ＋ Vite**：軽量SPA。
- **Convex React client**：`useQuery` / `useMutation` でリアクティブにデータ取得・更新。購読は自動。
- **MVP画面**：タスク一覧 ＋ シンプルなカンバン（作成・遷移・アサインのみ）。

---

## 8. 認証（単一テナント）

- ADR-5 により**マルチテナント不要**。チーム1つ分のメンバー認証で足りる。
- 候補：Convex Auth（OSS）。MVPは最小構成（メール＋パスワード or 単一OAuth）で開始。
- AIエージェントは認証ユーザーではなく **MCPトークン → Member** で表現（§5）。

---

## 9. テスト

- **Vitest**：リポジトリ規約（CLAUDE.md）に準拠。
- **convex-test**：Convex関数（mutation/query）の単体テスト。CLAUDE.md の方針どおり、**Convexバックエンドは外部依存としてモック対象**にできる（インメモリ実DBに依存しない）。状態遷移・採番などのロジックは振る舞い単位で検証。
- 実DB結合は結合テストで担保（CLAUDE.md のリポジトリ層テスト指針に整合）。

---

## 10. デプロイ／セルフホスト

- Convex は公式のセルフホスト用 Docker Compose（`backend` ＋ `dashboard` の2サービス）を同梱しており、自前インフラでホスト可能（**ADR-5 を満たす**）。
- 状態ストアはデフォルトSQLite、本番は外部Postgres/MySQLへ切替可能（いずれも Convex 内部の都合であり、アプリのデータモデルには影響しない）。
- 単一テナントのため、テナント分離・課金・オンボーディング機構は不要。
- **具体的なデプロイ構成（フロント／MCPサーバーの配置、compose統合の有無、永続ストレージ運用）は詳細設計・運用設計で決定する。** 本書では「Convexがセルフホスト可能」までを確定事項とする。

---

## 11. ロックインと移行戦略

- **懸念**：Convex 独自の関数モデル（query/mutation/action）への依存。
- **緩和**：
  - データモデルは論理モデル（§3）として保持済み。スキーマ移植は容易。
  - 不変条件を Convex functions に集約しているため、別バックエンドへ移す場合も「関数群の再実装」に閉じ、UI/MCPの公開インターフェースは保てる。
  - MCPサーバーは HTTP API ブリッジで疎結合。
