# 詳細画面設計：Issue 詳細 / Task 詳細

> 本書は `基本設計書.md`（§3 データモデル・§5 状態機械・ADR-9/10）を前提に、
> Issue と Task のクリック遷移で開く**詳細画面**の構成・遷移・バックエンド差分・実装順序を定義する。
> 対象は Phase 1 の閲覧機能。編集機能は Phase 2 として末尾に切り出す。

## 0. 設計判断ログ（本機能の ADR）

| # | 判断 | 採用 | 却下した代替案 | 理由 |
|---|---|---|---|---|
| ADR-D1 | **遷移方式** | URL ルーティング（`react-router-dom`） | モーダル／ドロワー、インラインパネル | 既存 `getByRef` が `{key}-{number}` 参照で引ける設計と一致。共有・ブックマーク・リロード耐性・ディープリンクが得られる。モーダルは URL を持てず状態がリロードで消える |
| ADR-D2 | **初期スコープ** | 閲覧優先（表示の完成）→ 編集は次段階 | 閲覧＋編集を一括実装 | レビュー単位を小さく保つ。更新系 mutation（`updateFields`/`transitionStatus`/`assign`/削除）は既に揃っており、Phase 2 で繋ぐだけ。競合 UX を編集段階に分離できる |
| ADR-D3 | **Markdown 描画** | `react-markdown` ＋ `remark-gfm` | 自前パーサ、プレーンテキスト表示 | `description` は §3 で Markdown と定義済み。安全な描画とGFM対応を定番ライブラリに委ねる |
| ADR-D4 | **URL の番号区切り** | パスは `/:projectKey/issues/:number`（Issue の表示は `Issue #5`、Task は `KEY-5`） | パスに `#` を含める | `#` は URL のフラグメント予約文字。パスでは種別（issues/tasks）でルートを分け、表示ラベルのみ `#`/`-` を使う。Issue ラベルにキーを含めないのは、Task の `{key}-{number}` との視認性を確保するため。プロジェクト横断で一意に参照する場面（MCP・コミットメッセージ等）では正準参照形式 `{key}#{number}` を使用 |

## 1. ルーティング設計

`react-router-dom` を導入し、`App.tsx` をルート構成へリファクタする（既存の1画面は `/` に退避）。

| パス | 画面 | 使用クエリ | not found 時 |
|---|---|---|---|
| `/` | 既存（プロジェクト選択＋Issue一覧＋カンバン） | 既存 | — |
| `/:projectKey/issues/:number` | Issue 詳細 | `issues.getByRef` | NotFound 表示 |
| `/:projectKey/tasks/:number` | Task 詳細 | `tasks.getByRef`（要拡張） | NotFound 表示 |

- `:number` は数値変換に失敗、または `getByRef` が `null` を返した場合 NotFound を表示する。
- 遷移元：既存 Issue 一覧の各行・`TaskCard` を `<Link>` 化する。
- 表示ラベルとパスの対応：Issue は `Issue #{number}` ⇔ `/KEY/issues/{number}`、Task は `KEY-{number}` ⇔ `/KEY/tasks/{number}`（ADR-D4）。

- 配下 tasks の各要素に `assigneeName` を付与（配下 Task 一覧の描画を簡素化）

### 4-3. フロント依存追加
- `react-router-dom`（ADR-D1）
- `react-markdown` ＋ `remark-gfm`（ADR-D3）

## 8. Issue一覧のフィルタリングとURL外在化（Issue #91）

一覧画面（Issue一覧・将来はBoard）に横断的に使う、フィルタ状態の設計方針を定める。

### 8-1. 設計原則（Linear分析より）

- **状態の外在化＝URL＝共有可能**: フィルタ状態はコンポーネントの `useState` に
  閉じず `URLSearchParams` に外在化する。URLを開き直す・共有する・ブックマーク
  するだけで同じ絞り込みが再現される。
- **語彙を閉じる**: フィルタ対象は固定属性（status/priority/assignee）のみとし、
  カスタムフィールドは持たない。UIの選択肢が既存定数（`ISSUE_STATUS_LABELS` /
  `PRIORITY_OPTIONS`）に限定されるため、URLを直接編集されても不正値は
  安全に無視できる（後述）。
- **単一文法**: 「属性を選ぶ→値を選ぶ」のみ。複数属性を指定した場合は**暗黙AND**
  とし、明示的なAND/OR切り替えやネストは行わない（個人〜小規模チームでは
  等値ANDで十分。段階的開示の後段として将来検討）。
- **不正値の無視**: URLを手動編集・共有時の劣化等で未知の値が来ても、
  パース関数（`parseFilterParams`）は既知集合に無い値を `null`（フィルタなし）に
  倒す。例外を投げたり空配列にフォールバックしたりして機能を壊さない。
- **filter と sort は別キーで直交**: フィルタ（`status`/`priority`/`assignee`）と
  ソート（`sort`/`dir`、#93）は別々の search params キーを使い、一方の
  `apply*Params` はもう一方のキーを温存する。相互に干渉しない。

### 8-2. 実装

- `src/lib/filterParams.ts`: `FilterState` の `parseFilterParams` /
  `applyFilterParams`（純粋関数、単体テスト対象）と、`useSearchParams`
  （react-router-dom）を薄くラップした `useFilterParams` フックを同居させる。
  `apply*` は既存の search params を複製し対象キーのみ `set`（`null` は
  `delete`）するため、`parse(apply(sp, state))` が `state` と一致する
  （往復一致・冪等性）。ソートの primitives（`SortState` /
  `parseSortParams` / `applySortParams`）も同ファイルに同居させ、#93（Issue一覧ソート）が消費する。
  さらに、filter と sort を同時に URL に書く統合型として、`applyListParams`
  （純粋関数）と `useIssueListParams`（フック）を追加した（Issue #98）。
- `src/components/FilterBar/FilterBar.tsx`: 「属性を選ぶ→値を選ぶ」の単一文法を
  ネイティブ `<select>` で実装した共有UI。どの属性を出すか（`attributes` prop）は
  呼び出し側が指定するため、Issue一覧とBoard（後続Issue）で異なる語彙を
  同一コンポーネントで扱える。担当者候補（`members`）は呼び出し側の購読済み
  一覧を props で受け取り、内部で二重購読しない。
- `IssuesView`: `issues.list` の返り値（派生ステータス・priority付き）を
  `useFilterParams` の状態でメモリ上フィルタする（クライアント側フィルタ。
  status/priorityにインデックスを張れない理由は基本設計書 Issue #91 参照）。
  `IssueStats`（俯瞰の分母）にはフィルタ前の全件、`IssueTable`
  （件数表示がフィルタ後件数に追従）にはフィルタ後を渡す。
- 担当者フィルタは Issue 配下の実行対象（canceled除外）Task の担当者集合
  （`issues.list` が返す `assignees: Id<"members">[]`）との包含判定で行う。

### 8-3. filter と display option（ソート）の直交（Issue #93）

Linear の原則に沿い、filter（**何を出すか**：status/priority/assignee による
絞り込み）と display option（**どう見せるか**：並び順）は独立した概念として
分離する。一方を変えても他方は保持され、互いの実装（コンポーネント・URL
キー・状態）に干渉しない。

- **URLキーの分離**: filter は `status`/`priority`/`assignee`、ソートは
  `sort`（対象フィールド: `priority` | `updatedAt`）/`dir`（`asc` | `desc`）
  というように、search params のキー空間を完全に分ける。`applyFilterParams`
  は `sort`/`dir` を、`applySortParams` は `status`/`priority`/`assignee` を
  互いに温存する（8-1 参照）。フィルタとソートを同時に URL 指定でき
  （例: `?status=open&sort=priority&dir=desc`）、片方の変更（select 操作）が
  もう片方のクエリを消さない。
- **コンポーネントの分離**: フィルタ UI は `FilterBar`、ソート UI は
  `SortBar`（別コンポーネント）に分離する。`FilterBar` は他Issue（#92）と
  並行利用中のため無改変で温存し、`SortBar` は独立ファイルとして追加する
  （見た目の作法＝ネイティブ `<select>` + 同一トークンは揃える）。
- **パイプラインの順序**: `IssuesView` では `issues.list` の返り値に対し
  「filter → sort」の順でメモリ上パイプラインを適用する（非破壊の
  `toSorted()` で並べ替え、購読配列・フィルタ結果は破壊しない）。sort が
  `null`（未指定）の場合はフィルタ結果をサーバー返却順のまま表示する。
- **URL書き込みの既知の制約と Issue #98 での対応**: search params ラッパは
  render スナップショットから URL を組み立てるため（react-router 仕様）、
  **同一バッチ内で複数の setter を呼ぶと後勝ちになり先の書き込みが失われる**。
  Issue #98 で、filter と sort を同時に書く導線に対応する `useIssueListParams()`
  フック（と純粋関数 `applyListParams`）を追加した。利用指針：filter/sort
  を同時に変更する場合は `useIssueListParams()` を使い、個別の setter の併用
  は避ける。`useSortParams` フックは廃止。`useFilterParams` は sort に触れない
  画面（TasksView/Board など）専用として温存。
- **priorityソートの重み付け**: priority は文字列（`"none" | "low" | ...`）
  のため辞書順比較では意味順にならない。`src/lib/taskMeta.ts` の
  `PRIORITY_WEIGHT`（`PRIORITY_OPTIONS` の並び順 none=0…urgent=4 から導出）
  で数値化してから比較する。`dir === "desc"` は比較関数の符号を反転して
  表現し（配列を昇順ソート後に `reverse()` する方式は同順位の相対順序が
  崩れるため使わない）、安定ソートを保つ。
- **Boardはスコープ外**: カンバン列内はユーザー手動 rank 順のままとし、
  ソートは提供しない（手動並べ替えとソートの二重定義を避け、列内の順序が
  常に予測可能であることを優先する）。
