# 基本設計書：AIネイティブ・タスク管理ツール（MVP）

> 本書は `タスク管理ツールコンセプト設計.md`（市場分析・ビジョン）を、実装可能な設計判断に落とし込んだものである。
> コンセプトが「何を目指すか」を語るのに対し、本書は「何を・どの順で・どう作るか」を定義する。

## 0. 設計判断ログ（ADR サマリ）

コンセプトを設計に変換するうえで確定した根本判断と、その理由・トレードオフを記録する。

| # | 判断 | 採用 | 却下した代替案 | 理由 |
|---|---|---|---|---|
| ADR-1 | **第一ターゲット** | 小規模チーム（2〜10人） | 個人＋AI／中〜大規模 | Linear 競合ゾーン。協調の価値は出るが、後述のとおり MVP では重い同期を避ける |
| ADR-2 | **真実の源泉（SSOT）** | クラウド同期バックエンド（マネージドBaaS／RDBMS いずれも可）が権威。Git は派生 | Git リポジトリを SSOT（Ghist/git-task 方式）／真の二重SSOT | 二重SSOTは同期問題が二乗化し破綻する。バックエンド一本化で同期をシンプルに保つ |
| ADR-3 | **「Gitネイティブ」の定義** | Git/GitHub ワークフローへの双方向バインディング（保存先ではなく連携先） | タスク本体を Markdown でリポジトリ保存 | ADR-2 と整合させるため。コンテキストスイッチ排除という価値は連携で達成可能 |
| ADR-4 | **MVPクサビ** | MCPサーバー＋最小タスク管理 | カンバン⇄ガント同期／AI主導スクラム | 差別化が最大・実装コスト最小・AIのコールドスタート問題を回避できる |
| ADR-5 | **ホスティング** | セルフホスト優先・単一テナント | SaaSマルチテナント／両対応 | テナント分離・課金・オンボーディングを不要化し MVP を最小化 |
| ADR-6 | **ステータス設計** | 固定6状態（オピニオネイテッド） | チームごとカスタマイズ可能 | 「設定の税金」を排除。Git自動遷移のマッピングを静的に保てる |
| ADR-7 | **同期方式** | サーバー権威＋楽観的UI＋リアクティブ購読／WebSocketプッシュ＋楽観的並行制御 | CRDTリアルタイム協調 | 2〜10人なら衝突は稀。CRDTは同期層の裏に隠し、後から差し替え可能にする |
| ADR-8 | **`.tasks/*.md` 投影** | Phase2 へ延期（GitLinkモデルは今設計） | MVPに同梱 | DB→Gitの投影同期とコンフリクト処理は MVP には重い |
| ADR-9 | **作業の階層** | `Project → Issue → Task` の3階層。Issue は解決すべき課題、Task はその解決手段 | Task をフラットに Project 直下に置く（Issue なし） | 「なぜこの作業をするのか（Issue）」と「何をするのか（Task）」を分離。AI/人間が課題単位で文脈を把握できる。Task は必ず Issue に従属（≥1対多） |
| ADR-10 | **Issue のステータス** | **Task 群からの派生**（独立した状態を持たない） | Issue 独自の状態機械を持つ | ADR-6（設定の税金排除）と同様、状態の二重管理と Task との乖離を避ける。Issue は常に ≥1 Task を持つため派生で一意に定まる |

## 1. プロダクト定義

**一行定義**：開発チームのタスクを Git ワークフローに密結合させ、人間と AI エージェント（Claude Code 等）が MCP を通じて対等に読み書きできる、セルフホスト型の最小タスク管理ツール。

**MVPで証明したい仮説**：
- AIエージェントが MCP 経由でタスクを自律的に読み書きできると、開発のコンテキストスイッチが実測で減る。
- commit / PR とタスク状態の自動同期で、二重入力がゼロになる。

**非ゴール（MVPでは検証しない）**：リアルタイム協調の一体感、ガントによる長期計画、AIの予測精度。

## 2. スコープ（MVP境界）

### IN（Phase 1 = MVP）
- データモデル：`Project / Member / Issue / Task / GitLink / Repository`
- 最小UI：タスク一覧 ＋ シンプルなカンバン（作成・ステータス遷移・アサイン・D&D並べ替え）。Issue 単位のグルーピング表示と派生ステータス表示を含む
- **MCPサーバー**：Resources（タスク・プロジェクト文脈）＋ Tools（CRUD・遷移・Gitリンク）、破壊的操作は Human-in-the-Loop 承認
- Git連携：GitHub Webhook ＋ commit メッセージ規約パーサによるステータス自動遷移
- セルフホスト：docker-compose（app + postgres）

### OUT（後回し。コンセプトの華やかな部分はほぼここ）
| 機能 | 投入フェーズ | 延期理由 |
|---|---|---|
| ガントチャート／カンバン⇄ガント同期 | Phase 2 | UXの目玉だが実装が重く、MVP仮説の検証に不要 |
| `.tasks/*.md` 投影（AIコンテキスト永続化） | Phase 2 | DB→Git投影同期とコンフリクト処理が重い |
| スプリント／ゴール／レトロボード | Phase 2–3 | スクラム支援はタスク管理が固まってから |
| **予測ベロシティ／AIキャパシティ計画** | Phase 3 | **コールドスタート問題**：ローンチ時に学習データが無い。データ蓄積後が正しい |
| CRDTリアルタイム協調 | Phase 3 | ADR-7。同期層の裏に隠し、必要になってから差し替え |
| SaaSマルチテナント | Phase 3 | ADR-5 |

## 3. データモデル（論理モデル・ストレージ非依存）

> 本節は**論理データモデル**であり、特定のストレージ技術（RDBMS／ドキュメント型リアクティブBaaS（Convex 等）／その他）に依存しない。
> 物理設計（テーブル定義・インデックス・トランザクション機構・リレーション表現）は**詳細設計に委ねる**。
> ここでは「何を保持し、どの不変条件を満たすか」を意味として定義し、それをどの機構（外部キー制約／ドキュメントID参照／プラットフォームのアトミック処理 など）で実現するかは問わない。
> 単一テナント前提のため、テナント識別子は持たない。

### 属性の意味型（実装機構に非依存）
| 意味型 | 定義 | 実装例（いずれも可） |
|---|---|---|
| Identifier | エンティティを一意に識別する値 | UUID／ULID／プラットフォーム発行ID |
| Reference | 別エンティティへの関連 | 外部キー／ドキュメントID参照／埋め込み |
| Enum | 有限の取りうる値の集合 | 文字列リテラル union／enum型 |
| Text | 自由記述（Markdown を含む） | — |
| Timestamp | 時刻 | — |
| OrderedRank | 一覧/ボード内の相対的並び順 | LexoRank 等の比較可能な値 |
| RevisionToken | 並行更新を検出するための改訂識別 | version整数／プラットフォームの楽観的並行制御トークン |

### エンティティと属性

**Project** — 作業の単位
- `id: Identifier`
- `key: Enum的短縮名`（例 "TASK" → 表示は `TASK-123`）。**一意**
- `name: Text` / `description: Text?`
- `name`, `createdAt: Timestamp`

**Member** — 人間およびAIエージェントの主体
- `id: Identifier`
- `name: Text` / `email: Text`（**一意**）
- `role: Enum { admin, member }`
- `createdAt: Timestamp`

**Issue** — 解決すべき課題（Task の上位概念、ADR-9）
- `id: Identifier`
- `project: Reference → Project`
- `number: 連番値`（プロジェクト内で一意）。**UI 表示は `Issue #{number}`**（語を付け Task の `{key}-{number}` と明確に区別）。MCP の正準参照は `{key}#{number}`（プロジェクト横断で一意に解決するため）
- `title: Text` / `description: Text?`（Markdown）
- `status: 派生 Enum { open, in_progress, done, canceled }`（**保持しない派生属性**。子 Task 群から算出。算出規則は §5.1）
- `createdBy: Reference → Member`
- `revision: RevisionToken`（並行更新検出）
- `createdAt / updatedAt: Timestamp`
- 不変条件：常に **≥1 個の Task** を持つ（最低 1対1、ADR-9 / INVARIANT-5）

**Task** — タスク本体（Issue を解決する手段）
- `id: Identifier`
- `issue: Reference → Issue`（**必須**。Task は必ずいずれかの Issue に従属する）
- `project: Reference → Project`（`issue.project` と一致させる冗長参照。プロジェクト単位のボード問い合わせを効率化するため保持。整合は Core ロジックで担保＝INVARIANT-5）
- `number: 連番値`（プロジェクト内で一意。表示は `{key}-{number}`）
- `title: Text` / `description: Text?`（Markdown）
- `status: Enum { backlog, todo, in_progress, in_review, done, canceled }`
- `priority: Enum { none, low, medium, high, urgent }`
- `assignee: Reference? → Member`
- `rank: OrderedRank`（カンバン並び順）
- `createdBy: Reference → Member`（人間／AIエージェントいずれも Member）
- `revision: RevisionToken`（並行更新検出）
- `createdAt / updatedAt: Timestamp`

**Repository** — Git連携先
- `id: Identifier`
- `project: Reference → Project`
- `provider: Enum { github }`
- `remoteUrl: Text`
- `webhookSecret: 機密値`（署名検証用。保存時は暗号化）
- `createdAt: Timestamp`

**GitLink** — タスクと Git アーティファクトの関連
- `id: Identifier`
- `task: Reference → Task` / `repository: Reference → Repository`
- `type: Enum { branch, commit, pull_request }`
- `externalRef: Text`（sha／PR番号／ブランチ名）
- `url: Text`
- `prState: Enum? { draft, open, merged, closed }`（`type=pull_request` のとき）
- `createdAt: Timestamp`

### 不変条件（INVARIANT）— 機構ではなく「保証すべき性質」として定義
1. **採番の一意性**：あるプロジェクト内で `Task.number` および `Issue.number` はそれぞれ一意。採番が同時実行されても重複しない。
   実現機構は問わない（アトミックなカウンタ更新／一意制約＋リトライ／プラットフォームのトランザクション など、詳細設計で選定）。
2. **並行更新の検出**：同一 Task／Issue への競合更新は失われず検出される。
   `RevisionToken` の比較で行う場合もあれば、BaaS のトランザクション／楽観的並行制御に委ねる場合もある（§4 参照）。
3. **参照整合性**：`Task.issue`、`Task.project`、`Issue.project`、`GitLink.task` 等の参照は実在するエンティティを指す。
4. **状態の妥当性**：`Task.status` の変化は §5 の状態機械が許す遷移のみ（保存層ではなく Core ロジックで強制）。`Issue.status` は派生属性であり、独立した遷移を持たない（§5.1）。
5. **Issue–Task の最低基数と整合**：各 Issue は常に **≥1 個の Task** を持つ（最低 1対1）。また `Task.project` は `Task.issue.project` と一致する。
   実現は Core ロジックで担保する（Issue 作成は最初の Task の作成を伴う／Issue の最後の Task を削除する操作は Issue ごと削除するか拒否する＝詳細設計で確定）。

### 設計上の注意
- **AIエージェントの主体**：エージェントも `Member`（`role=member`）として表現し、`Task.createdBy` に記録する。これにより「誰が（人間/AI）変更したか」の監査が可能。
- **拡張余地（Phase2）**：`Comment / ActivityLog`、`Sprint / Goal`、`.tasks/*.md` 投影状態（例 `projectionSyncedAt`）を、既存エンティティへの破壊的変更なしに追加できる粒度に保つ。

## 4. アーキテクチャ

### コンポーネント構成（セルフホスト・単一テナント）

```
┌──────────────────────────────────────────────────────────────┐
│ セルフホスト構成                                                │
│                                                                │
│  ┌────────────┐   リアクティブ購読   ┌──────────────────┐       │
│  │  Web UI    │◄───/WebSocket──────►│                  │       │
│  │ (React)    │   API呼び出し        │   Core ロジック   │       │
│  └────────────┘──────────────────►│  - 状態機械        │       │
│                                    │  - 採番           │       │
│  ┌────────────┐   内部API           │  - 権限/監査      │──────►┌────────────────┐
│  │ MCP Server │◄──────────────────►│  - Webhook受信    │       │  同期バックエンド │
│  │(TS/MCP SDK)│  stdio / HTTP      └──────────────────┘◄──────│ (BaaS / RDBMS) │
│  └────────────┘                            ▲                  └────────────────┘
│      ▲                                      │ GitHub Webhook
└──────┼──────────────────────────────────────┼─────────────────┘
       │ MCP (stdio/HTTP)                      │
  ┌────┴─────┐                          ┌──────┴──────┐
  │ Claude   │  ← AIエージェント          │   GitHub    │
  │ Code 等  │                          └─────────────┘
  └──────────┘
```

> **永続層は差し替え可能（ADR-2）**。「同期バックエンド」は、リアクティブBaaS（Convex 等：購読・トランザクション・並行制御をプラットフォームが提供）でも、自前の RDBMS でもよい。Core ロジックは**永続層の具体技術に依存しないインターフェース**越しにアクセスする。
> リアクティブBaaS を選ぶ場合、下図の「リアクティブ購読」「並行制御」はプラットフォーム機能で満たされ、Core/UI 側の実装が薄くなる。RDBMS を選ぶ場合は WebSocket プッシュ＋改訂トークン比較で同等の性質を実現する。

### 重要な設計原則
1. **MCPサーバーは永続層に直接触らない**。Core ロジック（内部インターフェース）経由でアクセスする。状態遷移規則・採番・権限・監査といった不変条件を一箇所に集約し、UI・MCP・Webhook の3経路すべてで同じ規則を強制するため。
2. **同期はサーバー権威（ADR-7）**。クライアントは楽観的に即時描画 → 内部で更新呼び出し → 成功で確定、競合で再取得。変更の伝播はリアクティブ購読（BaaS）または WebSocket ブロードキャスト（RDBMS構成）で行う。
3. **CRDTは同期層の裏に隔離**。UI とビジネスロジックは「変更イベント」抽象に依存させ、将来 CRDT へ差し替えてもデータモデル・公開APIが壊れないようにする。

### 並行制御（実装機構に非依存な要件）
不変条件は「同一タスクへの競合更新を失わず検出する」こと（§3 INVARIANT-2）。実現手段は永続層の選択に従う：
- **リアクティブBaaS の場合**：プラットフォームのトランザクション／楽観的並行制御に委ねる。
- **RDBMS の場合**：`RevisionToken` を更新条件に含め、不一致なら競合として扱い、クライアントは最新を再取得して「他の人が更新しました」を提示する。

## 5. ステータスワークフロー（固定6状態）

```
        ┌─────────┐
        │ backlog │
        └────┬────┘
             ▼
        ┌─────────┐      ┌──────────┐
        │  todo   │◄────►│ canceled │   ← canceled は任意状態から遷移可（破壊的＝承認要）
        └────┬────┘      └──────────┘
             ▼
      ┌─────────────┐
      │ in_progress │◄──────┐
      └──────┬──────┘       │ 差し戻し
             ▼              │
      ┌─────────────┐       │
      │  in_review  │───────┘
      └──────┬──────┘
             ▼
        ┌─────────┐
        │  done   │
        └─────────┘
```

- **前進**：backlog → todo → in_progress → in_review → done
- **差し戻し**：in_review → in_progress を許可。それ以外の任意逆行はバリデーションで拒否（規律維持）。
- **canceled**：任意状態から遷移可能だが**破壊的操作扱い**。MCP/自動化からの遷移は Human-in-the-Loop 承認を必須にする。
- 遷移規則は Core API の状態機械に単一実装し、UI・MCP・Webhook の全経路で共有する。

### 5.1 Issue の派生ステータス（ADR-10）

Issue は独立した状態機械を持たず、子 Task 群の状態から**一意に算出**する。状態の二重管理と Task との乖離を避けるため（純粋関数として Core ロジックに実装し、Task 遷移時に再計算する）。

算出規則（`active = canceled 以外の Task`）:

| Issue ステータス | 条件 |
|---|---|
| `canceled`（中止） | active が空（全 Task が canceled） |
| `done`（完了） | active が空でなく、active が**すべて** done |
| `in_progress`（着手中） | active のいずれかが `in_progress` / `in_review` / `done`（着手済みの Task が1つでもある） |
| `open`（未着手） | 上記以外（active がすべて `backlog` / `todo`） |

- Issue は常に ≥1 Task を持つ（INVARIANT-5）ため、上表は必ずいずれか1つに定まる。
- 「Issue が着手されているか」は `in_progress` で表現される。
- Git 駆動遷移（下表）は Task のみを対象とし、Issue ステータスはそれに伴って自動再計算される。

### Git駆動の自動遷移マッピング
| Gitイベント | 条件 | タスク遷移 |
|---|---|---|
| ブランチ作成 | ブランチ名に `TASK-123` を含む | todo → in_progress（任意・設定で無効化可） |
| commit | メッセージに `[TASK-123]` | GitLink(commit) を追加（遷移はしない） |
| PR open | 本文/タイトルに `TASK-123`、Draft | in_progress、GitLink(pull_request, draft) |
| PR ready for review | Draft 解除 | in_progress → in_review |
| PR merged | マージ | in_review → done |
| PR closed（未マージ） | クローズ | done以外なら in_progress に戻す |

- commit 規約：`[TASK-123]` をメッセージ本文の先頭または末尾で許容（正規表現 `\[([A-Z]+-\d+)\]`）。
- 自動遷移はあくまで規約に従った**前進方向**のみを行い、人間の明示操作を上書きしない設計（例：手動で done にしたものを PR open で in_progress に戻さない）。

## 6. MCPサーバー設計（本命の差別化）

Model Context Protocol サーバーを内蔵し、Claude Code 等が標準プロトコルでタスクを操作できるようにする。

### Resources（読み取り：LLMへ文脈を提供）
| URI | 内容 |
|---|---|
| `project://{key}` | プロジェクト概要・メンバー・アクティブ Issue 一覧（各 Issue の派生ステータス付き） |
| `issue://{key}/{number}` | Issue 全文（タイトル・説明・派生ステータス）＋ 配下 Task 一覧 |
| `task://{key}/{number}` | タスク全文（所属 Issue・タイトル・説明・状態・担当・GitLink・優先度） |
| `task://{key}/mine` | 呼び出し元エージェントに対応する Member の未完了タスク |

### Tools（実行：副作用あり）
| Tool | 引数 | 不変条件 | 承認 |
|---|---|---|---|
| `list_issues` | project, filter(status) | Issue ステータスは派生 | 不要 |
| `get_issue` | issue_ref | — | 不要 |
| `create_issue` | project, title, description?, first_task | 採番は Core API。最初の Task を必ず伴う（INVARIANT-5） | 不要 |
| `list_tasks` | project, filter(status/assignee/issue) | — | 不要 |
| `get_task` | task_ref | — | 不要 |
| `create_task` | issue_ref, title, description?, priority?, assignee? | 採番は Core API。issue 必須・project は issue から解決 | 不要 |
| `update_task` | task_id, fields, version | 楽観ロック | 不要 |
| `transition_status` | task_id, to_status | 状態機械で遷移検証。Issue ステータスを自動再計算 | done/canceled は要 |
| `assign_task` | task_id, member_id | — | 不要 |
| `link_git` | task_id, type, ref | Repository 紐付け確認 | 不要 |
| `delete_task` | task_id | Issue 最後の Task 削除時の扱いは INVARIANT-5 | **必須** |
| `delete_issue` | issue_ref | 配下 Task を併せて扱う（カスケード／拒否は詳細設計） | **必須** |

### Human-in-the-Loop（MCP仕様準拠）
- プロジェクトに重大な影響を与える操作（`delete_task`、`canceled` への遷移、一括操作）は、必ずホスト側で承認プロンプトを表示してから実行する。
- AIエージェントの全書き込みは `created_by`／監査ログに記録し、人間がレビュー・巻き戻し可能にする。

### アイデンティティ・マッピング（要設計）
- MCPアクセストークン → `Member` の対応付けで「AIエージェントが誰として動くか」を確定する。
- MVPでは「エージェント専用 Member」を1つ用意して運用し、Phase2で人間ユーザーへの委譲（on-behalf-of）を検討。

---

## 7. Git連携（GitHub Webhook）

- Repository ごとに `webhook_secret` を持ち、受信時に HMAC 署名（`X-Hub-Signature-256`）を検証する。
- 対象イベント：`push`（commit解析）、`pull_request`（open/ready/closed/merged）、`create`（branch）。
- リプレイ対策：イベントID（`X-GitHub-Delivery`）で冪等化し、重複処理を防ぐ。
- 解析失敗・未知タスク参照は**サイレントに握り潰さず**、ログに残し、必要なら通知する（CLAUDE.md のサイレント失敗回避方針に整合）。

## 9. 未解決リスク（次セッションで詰める）

1. **永続層の選定**：リアクティブBaaS（Convex 等）か RDBMS か。**ADR-5 のセルフホスト要件を満たすこと**が制約（例：Convex はセルフホスト可能なOSSバックエンドを持つ／マネージド専用サービスは ADR-5 と衝突する）。この選択が §3 INVARIANT の実現機構（採番・並行制御・購読）を左右する。
2. **MCPアイデンティティ**：エージェントの主体マッピングと on-behalf-of の設計。
3. **採番の一意性の実現**：高頻度作成時に INVARIANT-1 を満たす機構（プラットフォームのアトミック処理／一意制約＋リトライ／シーケンス等）を永続層に応じて決定。
4. **並行更新時のUX**：競合検出時のユーザー体験（フィールド単位で自動マージ可能なものの扱い）。
5. **Webhook信頼性**：到達保証なし。定期 reconcile（GitHub APIポーリング）でズレを補正するか。
6. **AI書き込みの安全性**：監査ログの粒度と巻き戻し（undo）の実装範囲。
7. **スタック確定**：Core ロジックの実行形態、フロント、MCP SDK バージョン（永続層選定後に確定）。
