# UI文言・配置規約

> 本書は UI に表示する全文言（ボタン・見出し・ナビ・エラー・空状態・確認・
> aria-label・placeholder）の用語統一と、作成・削除などのアクション導線の配置を
> 定める。表記ゆれ（「タスク」「課題」「イシュー」の混在、「追加」「新規」
> 「作成」の混在など）と配置の非対称（作成・削除の導線がエンティティごとに
> 違う場所にある等）による認知負荷の解消が目的。対象読者は `src/` の UI に
> 触れる開発者。

---

## 1. 目的と適用範囲

対象は次のすべて。

- ボタン・見出し・ナビゲーション
- エラー文言・空状態文言・確認ダイアログ文言
- `aria-label` / `placeholder`（スクリーンリーダー・入力補助のためのテキストも
  画面表示文言と同じ扱いとする）
- 上記に関連するアクション導線（作成・削除・編集・状態遷移・「戻る」）の
  置き場所

本書が扱わない領域は次の既存ドキュメントを正とする。

- CSS の実装規約（レイヤー・トークン・CSS Modules）は
  [フロントエンドCSS規約.md](./フロントエンドCSS規約.md)。
- ID の表示形式（Issue は `Issue #5`、Task は `KEY-5`）は
  [詳細画面設計.md](./詳細画面設計.md) §0 ADR-D4。本書のエンティティ名統一
  （§2）とこの表示形式は矛盾しない。`formatIssueRef` は既に
  `` `Issue #${number}` `` を返しており（`src/lib/formatIssueRef.ts:7`）、
  ADR-D4 の表示形式は英語表記のエンティティ名と整合している。

判断の上位原則（認知負荷理論・行動心理学に基づく汎用 UI/UX 原則）は
開発環境側の `ui-ux` Skill が定める。本書はその task-manager 固有の具体化であり、
両者が食い違う場合は本書（プロジェクト固有規約）を優先する。

## 2. エンティティ名

| 概念 | 統一表記 | 禁止表記 |
|---|---|---|
| Issue（課題） | `Issue`（英語固定） | 課題／イシュー |
| Task（解決手段） | `Task`（英語固定） | タスク |

UI 全体は日本語のままとし、動詞・メタラベル（「作成」「削除」「優先度」等）は
日本語で書く。エンティティ名だけを英語に固定し、「Task を削除」のように
分かち書きする。対象はボタン・見出し・ナビだけでなく、aria-label・
placeholder・エラー文言・空状態文言も含む（例:「タスクのタイトル」→
「Task のタイトル」）。

## 3. 動詞ラベル統一表

| 場面 | 統一語 | 是正前の分裂例（解消済み） |
|---|---|---|
| 新規作成・起動ボタン | 「＋ Issue を作成」／「＋ Task を作成」（幅制約時は「＋ 作成」でも可） | `NewIssueForm.tsx`「＋ 新規 Issue」／`AddTaskForm.tsx`「＋ タスク」——いずれも不適合だったが #103 で統一済み |
| 新規作成・確定ボタン | 「作成」 | `NewIssueForm.tsx`「作成」（元々適合）／`AddTaskForm.tsx`「追加」（不適合だったが #103 で「作成」に統一済み） |
| 新規作成・失敗時エラー | 「作成に失敗しました」 | `NewIssueForm.tsx`（元々適合）／`AddTaskForm.tsx`「追加に失敗しました」（不適合だったが #103 で「作成に失敗しました」に統一済み） |
| 中断 | 「キャンセル」 | `NewIssueForm.tsx` / `DetailEditForm.tsx` / `ConfirmPanel.tsx`（元々適合）／`AddTaskForm.tsx`「取消」（不適合だったが #103 で「キャンセル」に統一済み） |
| 削除・起動ボタン | 「Issue を削除」／「Task を削除」（配置は §5） | `TaskDetail.tsx`「タスクを削除」はエンティティ名が §2 に反していたが #104 で「Task を削除」へ修正済み。`IssueTable.tsx`（旧・行内）の素の「削除」ボタンは #105 で撤去済み（削除導線は詳細画面の danger セクションのみ） |
| 削除・確認ボタン | 「削除する」 | `TaskDetail.tsx`（元々適合）。`IssueTable.tsx`（旧・行内）の確認パネルは #105 で撤去済み（削除導線は詳細画面の danger セクションのみ） |

「追加」「新規」「取消」は廃止語彙。新規に文言を書く際にこれらを使わない。

## 4. ステータス・優先度ラベル対応表

ラベル定数の正は `src/lib/issueMeta.ts`（`ISSUE_STATUS_LABELS`）と
`src/lib/taskMeta.ts`（`TASK_STATUS_LABELS` / `PRIORITY_LABELS`）のみとする。
**コンポーネント内での再定義を禁止する**。是正前は `TaskCard.tsx` が
`PRIORITY_LABELS` を `taskMeta.ts` と別に再定義しており、`none` の表示が
食い違っていた（TaskCard は「—」、taskMeta 側は「なし」）。#102 で
`taskMeta.ts` の定義（`PRIORITY_OPTIONS` から導出される単一ソース）に
一本化し、`none` の表示は「なし」に寄せた（重複排除は挙動不変の機械的作業
ではなく、表示をどちらへ寄せるかの決定を伴っていた）。

### 4-1. ステータス

| 内部値 | 統一ラベル | Issue 是正前（`issueMeta.ts`） | Task 是正前（`taskMeta.ts`） |
|---|---|---|---|
| `open` / `todo` | 未着手 | 未着手 | 未着手 |
| `in_progress` | **進行中** | 着手中 → #102 で「進行中」に統一済み | 進行中（変更なし） |
| `done` | 完了 | 完了 | 完了 |
| `canceled` | **中止** | 中止（変更なし） | キャンセル → #102 で「中止」に統一済み |
| `backlog`（Task のみ） | バックログ | — | バックログ |
| `in_review`（Task のみ） | レビュー中 | — | レビュー中 |

`canceled` を「キャンセル」ではなく「中止」に統一するのは、ボタンの
「キャンセル」（§3・中断の意）と語が衝突するため。1つの語に1つの概念だけを
割り当て、「キャンセルする」がボタン操作を指すのか状態遷移を指すのか読み手が
迷わないようにする。

副作用として `ActiveIssueStrip.tsx` の「進行中の Issue はありません。」は
是正前の `issueMeta.ts` の「着手中」ラベルと矛盾していたが、#102 の統一により
「進行中」で一致している。

### 4-2. 優先度

優先度ラベルは元々 `taskMeta.ts` の `PRIORITY_OPTIONS` に一本化されており
Issue/Task で分裂していない。`TaskCard.tsx` の再定義は #102 で解消済み（上記）。

## 5. ボタン配置規約

| 導線 | 置き場所 | 根拠 |
|---|---|---|
| Issue 作成 | Issue 一覧（`/issues`）最上部 | 既存どおり（`IssuesView.tsx:83-85`）。Board 画面（`/`）には #106 で `/issues` への誘導リンクを追加済み（空状態文言内、`Board.tsx:420-427`）。**Board 画面にインラインの Issue 作成フォームは置かない**——D&D の in-flight mutation 中に別経路の mutation が columns を書き換え楽観更新の巻き戻りを誘発するため原則採用しない（採用する場合は実機6シナリオ×フィルタ ON/OFF に「ドロップ直後の in-flight 中に Issue 作成」を加えた実機検証が必須） |
| Task 作成 | 親 Issue 詳細画面のタスクセクション末尾 | 既存どおり（`IssueDetail.tsx:252-259`）。Task は Issue に従属するため、作成は親 Issue の文脈内で行う |
| 削除 | 詳細画面最下部の danger セクションに統一 | 破壊的操作の隔離によるエラー防止と、Issue/Task 間の対称性の確保。Task は元々 `TaskDetail.tsx:467-485` に dangerSection を持つ。Issue には #104 で danger セクションを新設し（`IssueDetail.tsx:277-296`）、#105 で一覧行末の削除ボタン・確認パネル（旧 `IssueTable.tsx`）を撤去済み。削除導線は詳細画面の danger セクションのみ |
| 編集 | 詳細ヘッダ右端 | 現状維持（`IssueDetail.tsx:163-171` / `TaskDetail.tsx:302-310`） |
| 戻る（← 一覧へ） | そのエンティティの一覧へ | Issue 詳細は `/issues` へ、Task 詳細は `/` へ。#104 で修正済み（是正前は両方 `to="/"` 固定で、Issue 詳細から戻ると Issue 一覧ではなく Board に着地していた）。`/issues` も `/` 同様 `AppLayout` 配下の子ルート（`src/App.tsx:16-39`）のため、プロジェクト選択の sessionStorage 復元（`AppLayout.tsx:14-17`）は変更後も維持される |
| 状態遷移 | Task 詳細のみ | 既存どおり（`TaskDetail.tsx:386-398`）。Issue のステータスは配下 Task から自動算出される派生値（基本設計書.md §5.1「Issue の派生ステータス（ADR-10）」。実装は `convex/lib/issueStatus.ts` の `deriveIssueStatus`）のため、Issue 詳細には遷移ボタンを置かず「ステータスは配下 Task から自動算出されます」という説明文を置く（#104 で設置済み、`IssueDetail.tsx:173-175`） |

## 6. 文言パターン

### 6-1. 確認ダイアログ（宣言形に統一）

「この {対象} を削除します。{付随して消えるもの}。取り消せません。」＋
確認ボタン「削除する」／中断ボタン「キャンセル」の宣言形に統一する。疑問形
（「〜しますか？」）は使わない。

- 適合例（Issue 削除）: `IssueDetail.tsx:291`「この Issue と配下の Task・
  Git 連携をすべて削除します。取り消せません。」——#104 で Issue 詳細に
  新設した danger セクションの文言。是正前は一覧行内（旧 `IssueTable.tsx`）に
  同種の確認パネルがあり「配下のタスク」と §2 違反の表記だったが、#105 で
  導線ごと撤去済み（削除導線は詳細画面の danger セクションに一本化）
- 適合例（Task 削除）: `TaskDetail.tsx:480`「この Task を削除します。関連する
  Git 連携も併せて削除されます。取り消せません。」——是正前は「このタスクを
  削除しますか？関連する Git 連携も併せて削除されます。」という疑問形＋
  §2 違反のエンティティ名だったが、#104 で宣言形・エンティティ名とも
  修正済み
- 適合例（状態遷移）: `TaskDetail.tsx:448`「「{ステータス}」へ遷移します。
  この操作は取り消せません。」（是正前から適合）

### 6-2. エラー文言

「{操作}に失敗しました」に統一する。`NewIssueForm.tsx:36`「作成に失敗しました」
／`AddTaskForm.tsx:32`「作成に失敗しました」（是正前は「追加に失敗しました」で
不適合だったが #103 で統一済み）。ただし複数操作をまとめて扱う汎用フォールバック
（`Board.tsx:68` / `TaskDetail.tsx:231` の「操作に失敗しました」、いずれも D&D や
状態遷移・削除・担当変更を一括で受ける catch 節）は個別の操作名に分解できないため
対象外とする。

### 6-3. 空状態

次に取るべき行動を、**実在する導線の名前**で案内する。是正前の `Board.tsx`
（`serverIsEmpty` の空状態文言）は「タスクがありません。Issue 一覧の
「＋ タスク」または「＋ 新規 Issue」から作成できます。」で、「＋ タスク」は
Board 画面（`/`）上には存在せず、実際には Issue 詳細にしかない導線を指しており
案内どおりにはたどり着けない反面教師だった。#106 で「Task がありません。
Issue 一覧から Issue を作成してください。」＋ `/issues` への誘導リンク
（`Board.tsx:420-427`）という、Board 画面上で完結し実在する導線だけを案内する
形に修正済み。案内文言を書く際は、指す導線が同一画面または遷移先に実在する
ことを確認する。

## 7. 運用

- UI 文言・アクション導線を追加／変更する際は本書に従う。着手前に
  `ui-ux` Skill（汎用原則）と本書を読み込むこと。
- UI 文言・導線に触れる変更のレビューでは、本書準拠を確認観点に含める
  （frontend-specialist の領域レビュー観点）。
- §2〜§6 に記録した是正前の分裂例は #102〜#106 で解消済み。新規の文言・
  導線を追加・変更する際は本書の統一表・配置ルールに従う。
- 将来課題: UI 文言は現状コンポーネントへ直書きされており定数化されていない
  （`ISSUE_STATUS_LABELS` / `TASK_STATUS_LABELS` / `PRIORITY_LABELS` 以外の
  ボタン・見出し・エラー文言など）。本書の統一表を元に文言定数
  （`uiText.ts` 等）への一元化を検討する。
